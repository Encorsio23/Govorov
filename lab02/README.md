Проблема: В приложении с загрузкой тяжелых файлов, как видео, возникает проблема стоимости ресурсов и больших затрат трафика на загрузку

Цель: Разработать видеоплеер (мини ютуб)

Применение паттерна Proxy:

Интерфейс IVideo (Subject): Общий «контракт» для видео и его заместителя. Содержит метод Play(). Благодаря ему приложению всё равно, работает оно с реальным файлом или с прокси.
Класс RealVideo (RealSubject): Тяжелый класс. В его конструкторе прописана логика подключения к сети и загрузки потока. Это именно тот объект, создания которого мы хотим избежать до последнего момента.
Класс VideoProxy (Proxy): «Умный секретарь». Содержит ссылку на RealVideo. При вызове метода Play() он сначала проверяет: «А создано ли у меня реальное видео?». Если нет — создает его, если да — просто передает команду дальше.
Класс AppUI (Client): Работает со списком объектов IVideo. Приложение «думает», что у него уже есть все видео, хотя на самом деле это лишь легкие «заместители».
